## 4. 테스트 케이스

## 4.1 홈(프로파일) 페이지

### TC-HOME-001: 홈 페이지 기본 렌더링
- 목적: 홈 진입 시 주요 섹션이 정상 표시되는지 확인
- 사전조건: 서버 정상 동작
- 절차:
  1) / 접속
  2) Hero/TechStack/Timeline/CTA 표시 확인
- 기대결과:
  - Hero 텍스트가 렌더링됨
  - Tech Stack 배지가 노출됨
  - CTA 버튼(프로젝트 보기/연락하기) 노출됨

### TC-HOME-002: 반응형 레이아웃(모바일)
- 목적: 모바일 화면에서 레이아웃 깨짐 없이 표시되는지 확인
- 절차:
  1) 모바일 해상도(375x812)로 / 접속
  2) 섹션 간격/폰트/버튼 터치 영역 확인
- 기대결과:
  - 가로 스크롤이 발생하지 않음
  - 버튼은 최소 터치 영역 확보(손가락 탭 가능)

### TC-HOME-003: 다크모드 전환
- 목적: 다크모드 토글 또는 시스템 다크모드 연동 확인
- 절차:
  1) 라이트 모드에서 / 확인
  2) 다크모드로 전환
  3) 텍스트 대비/가독성 확인
- 기대결과:
  - 배경/텍스트/카드 대비가 유지됨
  - 아이콘 및 링크가 식별 가능함

## 4.2 프로젝트 리스트/상세

### TC-PROJ-001: 프로젝트 리스트 조회
- 목적: 프로젝트 목록이 정상 로딩되는지 확인
- 절차:
  1) /projects 접속
  2) 카드가 1개 이상 노출되는지 확인
- 기대결과:
  - 프로젝트 카드가 렌더링됨
  - 제목/요약/태그가 노출됨

### TC-PROJ-002: 프로젝트 태그 필터
- 목적: 태그 필터 적용 시 목록이 올바르게 갱신되는지 확인
- 절차:
  1) /projects 접속
  2) 태그 AI 선택
- 기대결과:
  - AI 태그 포함 프로젝트만 노출됨
  - 결과 개수 표시가 있다면 함께 갱신됨

### TC-PROJ-003: 프로젝트 정렬(최근순)
- 목적: 정렬 옵션이 정상 동작하는지 확인
- 절차:
  1) /projects 접속
  2) 정렬을 최근순으로 변경
- 기대결과:
  - createdAt 기준 내림차순 정렬됨

### TC-PROJ-004: 프로젝트 상세 페이지 이동
- 목적: 상세 페이지 라우팅 및 콘텐츠 렌더링 확인
- 절차:
  1) /projects에서 임의 카드 클릭
  2) /projects/{slug} 이동 확인
- 기대결과:
  - URL이 slug 기반으로 변경됨
  - 본문/태그/링크가 표시됨

### TC-PROJ-005: 존재하지 않는 slug 처리
- 목적: 잘못된 slug 접근 시 404 처리 확인
- 절차:
  1) /projects/not-exist-slug 접속
- 기대결과:
  - 404 페이지 또는 Not Found UI 노출
  - 콘솔 에러가 과도하게 발생하지 않음

## 4.3 블로그 리스트/상세

### TC-BLOG-001: 블로그 리스트 로딩
- 목적: 블로그 글 목록이 정상 표시되는지 확인
- 절차:
  1) /blog 접속
- 기대결과:
  - 글 카드/리스트가 노출됨
  - 제목/요약/태그/날짜 노출

### TC-BLOG-002: 블로그 상세 렌더링(TOC 포함)
- 목적: 본문 및 TOC가 정상 생성되는지 확인
- 절차:
  1) /blog/{slug} 접속
  2) TOC 노출 확인
  3) 헤딩 클릭 시 스크롤 이동 확인
- 기대결과:
  - TOC가 헤딩 구조대로 생성됨
  - 헤딩 클릭 시 해당 위치로 이동됨

### TC-BLOG-003: 코드 블록 하이라이팅
- 목적: 코드 하이라이트가 정상 표시되는지 확인
- 절차:
  1) 코드 블록 포함 글 접속
- 기대결과:
  - 코드가 가독성 있게 렌더링됨
  - 긴 코드에서 가로 스크롤이 필요하면 정상 제공됨

### TC-BLOG-004: 읽기 진행 바(있을 경우)
- 목적: 스크롤 시 진행 바가 정상 업데이트되는지 확인
- 절차:
  1) 긴 글 접속
  2) 스크롤 이동
- 기대결과:
  - 진행 바가 0% → 100%로 자연스럽게 변화

## 4.5 Contact(연락) 기능

### TC-CNT-001: Contact 폼 유효성 검증
- 목적: 필수 입력값 검증 확인
- 절차:
  1) /contact 접속
  2) 빈 값으로 제출
- 기대결과:
  - 필수 항목 오류 메시지 표시
  - 제출이 차단됨

### TC-CNT-002: Contact 정상 제출
- 목적: Contact 저장/API 연동 확인
- 절차:
  1) 유효한 name/email/message 입력
  2) 제출
- 기대결과:
  - 성공 토스트/메시지 표시
  - Backend에 데이터가 저장됨

### TC-CNT-003: Rate Limit 동작
- 목적: 과도한 요청 시 제한되는지 확인
- 절차:
  1) 짧은 시간에 연속 제출(예: 10회)
- 기대결과:
  - 일정 횟수 이후 429 또는 제한 안내 표시
  - UX는 친절한 오류 메시지 제공

## 4.6 Backend API 테스트

### TC-API-001: 프로젝트 목록 API
- 목적: /api/projects 반환 형식 검증
- 절차:
  1) GET /api/projects 호출
- 기대결과:
  - 200 응답
  - 배열 형태 데이터 반환
  - 각 항목에 title/slug/summary/tags 포함

### TC-API-002: 프로젝트 상세 API
- 목적: /api/projects/{slug} 반환 검증
- 절차:
  1) GET /api/projects/project-a 호출
- 기대결과:
  - 200 응답
  - content(본문) 포함

### TC-API-003: 블로그 목록/상세 API
- 목적: /api/blog 및 /api/blog/{slug} 동작 확인
- 절차:
  1) GET /api/blog
  2) GET /api/blog/post-a
- 기대결과:
  - 200 응답
  - publishedAt 및 tags 포함

### TC-API-004: 존재하지 않는 리소스
- 목적: 잘못된 slug 요청 처리 확인
- 절차:
  1) GET /api/projects/not-exist 호출
- 기대결과:
  - 404 응답 또는 명확한 에러 코드
  - 에러 메시지는 내부 정보 노출 없이 최소화

## 4.9 Flutter WebView 테스트

### TC-WV-001: WebView 초기 로딩
- 목적: 앱 진입 시 웹이 정상 로딩되는지 확인
- 절차:
  1) Flutter 앱 실행
  2) WebView 첫 화면 확인
- 기대결과:
  - 빈 화면 없이 로딩 인디케이터 처리(구현 시)
  - 홈 화면이 정상 표시됨

### TC-WV-002: 내부 라우팅 이동
- 목적: WebView 내 페이지 이동이 정상인지 확인
- 절차:
  1) 프로젝트 상세로 이동
  2) 뒤로가기(시스템 백) 수행
- 기대결과:
  - WebView history 기반 뒤로가기 정상
  - 앱이 종료되지 않고 이전 페이지로 이동

### TC-WV-003: 외부 링크 처리
- 목적: GitHub/외부 사이트 링크 클릭 시 처리 확인
- 절차:
  1) GitHub 링크 클릭
- 기대결과:
  - 정책에 따라 외부 브라우저로 열거나(권장) WebView에서 새 창 처리

### TC-WV-004: Safe Area 확인(iOS)
- 목적: 노치/홈 인디케이터 영역 겹침 방지 확인
- 절차:
  1) iPhone 노치 디바이스에서 실행
- 기대결과:
  - 상단/하단 UI가 안전영역을 침범하지 않음
