# USE CASE — 개발자 프로파일 블로그 & 앱 확장(WebView) 플랫폼

## 1. 문서 개요
- 목적: 시스템 사용자 관점에서 주요 기능의 사용 흐름과 상호작용을 정의
- 범위: Web(Next.js), Backend(Node.js), Flutter(WebView)
- 버전: v1.0

---

## 2. 사용자 정의 (Actors)

### 2.1 Actor 목록
- Visitor (방문자)
  - 사이트를 방문해 정보를 탐색하는 일반 사용자
- Recruiter (채용 담당자)
  - 개발자 역량/프로젝트/이력을 확인하려는 사용자
- Collaborator (협업 제안자)
  - 프로젝트 협업 또는 외주를 문의하려는 사용자
- Owner (사이트 소유자/개발자)
  - 콘텐츠를 작성·관리하는 운영자

---

## 3. Use Case 다이어그램 개요(텍스트)

Visitor / Recruiter / Collaborator  
→ 프로파일 조회  
→ 프로젝트 탐색  
→ 블로그 열람  
→ 연락(Contact)

Owner  
→ 콘텐츠 작성/관리  
→ 배포 및 운영

---

## 4. Use Case 상세

## UC-01: 홈(프로파일) 조회

- Actor: Visitor, Recruiter, Collaborator
- 목적: 개발자에 대한 전체적인 소개 및 방향성 파악
- 사전조건: 서비스가 정상적으로 배포되어 있음
- 트리거: 사용자가 메인 URL 접속

### 기본 흐름
1. 사용자가 메인 페이지에 접속한다.
2. 시스템은 Hero 영역을 표시한다.
3. 기술 스택, 타임라인, CTA 버튼을 노출한다.

### 대안 흐름
- A1: 느린 네트워크 환경
  - Skeleton UI 표시 후 콘텐츠 렌더링

### 결과
- 사용자는 개발자의 전문 영역을 빠르게 이해한다.

---

## UC-02: 프로젝트 목록 탐색

- Actor: Visitor, Recruiter
- 목적: 개발자가 수행한 프로젝트를 개괄적으로 파악
- 사전조건: 프로젝트 데이터가 존재함
- 트리거: 사용자가 “프로젝트” 메뉴 선택

### 기본 흐름
1. 시스템은 프로젝트 리스트를 카드 형태로 표시한다.
2. 사용자는 태그 또는 정렬 옵션을 선택한다.
3. 리스트는 선택 조건에 맞게 갱신된다.

### 대안 흐름
- A1: 필터 결과 없음
  - “조건에 맞는 프로젝트가 없습니다” 메시지 표시

### 결과
- 사용자는 관심 있는 프로젝트를 선별할 수 있다.

---

## UC-03: 프로젝트 상세 조회

- Actor: Visitor, Recruiter, Collaborator
- 목적: 특정 프로젝트의 문제 해결 방식과 성과 이해
- 사전조건: 유효한 프로젝트 slug 존재
- 트리거: 프로젝트 카드 클릭

### 기본 흐름
1. 시스템은 프로젝트 상세 페이지로 이동한다.
2. 문제 정의, 해결 방법, 사용 기술, 결과를 표시한다.
3. 외부 링크(GitHub, Demo)를 제공한다.

### 예외 흐름
- E1: 존재하지 않는 slug 접근
  - 404 페이지 표시

### 결과
- 사용자는 개발자의 실무 능력을 구체적으로 파악한다.

---

## UC-04: 블로그 글 탐색

- Actor: Visitor, Recruiter
- 목적: 개발자의 사고 과정과 기술적 깊이 파악
- 사전조건: 블로그 글이 1개 이상 존재
- 트리거: “블로그” 메뉴 선택

### 기본 흐름
1. 시스템은 블로그 글 목록을 표시한다.
2. 사용자는 특정 글을 선택한다.
3. 시스템은 본문과 TOC를 렌더링한다.

### 결과
- 사용자는 기술적 인사이트를 얻는다.

---

## UC-05: 블로그 글 검색

- Actor: Visitor, Recruiter
- 목적: 관심 주제의 글을 빠르게 찾기
- 사전조건: 검색 인덱스가 구성되어 있음
- 트리거: 검색창에 키워드 입력

### 기본 흐름
1. 사용자가 키워드를 입력한다.
2. 시스템은 관련 글/프로젝트를 검색한다.
3. 결과 리스트를 표시한다.

### 대안 흐름
- A1: 검색 결과 없음
  - “검색 결과 없음” 메시지 표시

### 결과
- 사용자는 원하는 정보를 빠르게 찾는다.

---

## UC-06: 연락(Contact) 요청

- Actor: Recruiter, Collaborator
- 목적: 채용/협업/문의 전달
- 사전조건: Contact API 정상 동작
- 트리거: “연락하기” 버튼 클릭

### 기본 흐름
1. 사용자가 Contact Form을 연다.
2. 이름, 이메일, 메시지를 입력한다.
3. 제출 버튼을 클릭한다.
4. 시스템은 입력값을 검증 후 저장한다.
5. 성공 메시지를 표시한다.

### 예외 흐름
- E1: 필수 입력 누락
  - 오류 메시지 표시
- E2: Rate Limit 초과
  - 제한 안내 메시지 표시

### 결과
- 운영자는 문의 내용을 확인할 수 있다.

---

## UC-07: Flutter WebView 앱 이용

- Actor: Mobile User
- 목적: 앱 형태로 동일한 콘텐츠 이용
- 사전조건: Flutter 앱 설치
- 트리거: 앱 실행

### 기본 흐름
1. 앱 실행 시 WebView가 로드된다.
2. 홈 페이지가 표시된다.
3. 사용자는 웹과 동일하게 탐색한다.
4. 시스템 뒤로가기 시 WebView 히스토리가 적용된다.

### 결과
- 사용자는 네이티브 앱처럼 콘텐츠를 소비한다.

---

## UC-08: 콘텐츠 관리(운영자)

- Actor: Owner
- 목적: 프로젝트/블로그 콘텐츠 유지 및 확장
- 사전조건: 로컬 개발 환경 또는 CMS 접근
- 트리거: 콘텐츠 수정 필요 발생

### 기본 흐름
1. 운영자는 MDX 파일을 수정한다.
2. 변경 사항을 커밋한다.
3. CI/CD를 통해 자동 배포된다.

### 결과
- 최신 콘텐츠가 사용자에게 제공된다.

---
